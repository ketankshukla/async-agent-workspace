# Resume Project — Async Agent Workspace

## [Async Agent Workspace](https://async-agent-workspace.vercel.app/) ([GitHub](https://github.com/ketankshukla/async-agent-workspace))

*July 2026*

- Designed and built a full-stack durable background-agent platform from scratch using **Next.js 16 (App Router), TypeScript, and Tailwind CSS**, deployed as a production web application on Vercel, that lets a user submit a research task and watch a Claude-powered agent execute it asynchronously with progress streaming live to the browser.
- Architected the system's core reliability pattern around **Inngest**, a durable execution engine: every unit of agent work — marking a run as running, generating a plan, each Claude call, each tool execution, and finalizing — is wrapped in its own independently checkpointed `step.run(...)` call, so a crash or transient failure retries only the failed step instead of restarting a multi-minute task from scratch.
- Implemented the Claude **tool-use (function-calling)** loop as a decoupled, dependency-injected pure function, separate from Inngest's own step engine — accepting an abstract "step-like" runner interface — specifically so the agentic loop's iteration-cap termination behavior could be unit-tested against a mocked model client, without needing to simulate Inngest's actual execution runtime.
- Built a `fetch_url` tool using **Cheerio** for HTML-to-text extraction (stripping scripts, styles, navigation, and footers), enforcing a request timeout and truncating output to ~6,000 characters, and validating URL schemes before any network call is made — all covered by tests that assert the failure paths never touch the network.
- Designed a **Postgres schema** (`runs`, `run_steps`) in **Supabase** with row-level security enabling public read-only access while restricting all writes to a server-only service-role client — enforcing the security boundary between browser-safe reads and privileged background-job writes at the architecture level, not merely by convention.
- Implemented live UI updates end-to-end with **Supabase Realtime**: subscribed React components to `postgres_changes` events on both tables (step updates filtered by run ID), so every plan, tool-call, tool-result, and final-answer step the agent writes appears in the browser instantly with zero polling, and full run history correctly reloads after closing and reopening the tab.
- Engineered safety limits directly into the agent loop — a **hard cap of 6 tool-use iterations** to bound runaway execution and API cost — plus explicit failure-state handling so a run that fails to enqueue or fails mid-execution is marked `failed` with a human-readable error message rather than left silently stuck.
- Wrote a **Vitest** unit test suite (12 tests) covering the `fetch_url` tool's error handling (invalid URLs, timeouts, non-OK HTTP responses) with zero live network calls, the agent loop's iteration-cap termination behavior against a mocked Claude client, and API input validation for the task-submission endpoint.
- Set up a **GitHub Actions CI pipeline** (lint, test, build on every push/PR) and diagnosed and fixed a subtle Next.js build-time issue where server-only API route modules are imported during static page-data collection even for fully dynamic routes — resolved by supplying non-secret placeholder environment variables in CI rather than live credentials.
- Deployed to **Vercel via its native GitHub integration** and connected **Inngest's Vercel Marketplace integration** for production background-job execution, so every merge to `master` auto-deploys and automatically re-syncs the background function.
- Authored complete project documentation — README, user guide, and a technical architecture deep-dive with a full request-flow diagram and data model — tracing the enqueue → durable job → checkpointed steps → live UI pipeline for future maintainers.
